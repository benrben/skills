Reading the graph

Each card is a module
fill = depth tier · how hardened a module is
deep foundations sink to the bottom; entrypoints rise to the top
bottom bar = test coverage
ring + ! = the agent has a proposal
blue outline = changed since last scan
red dashed = a dependency leak
◇ dashed card = planned (intended) module
striped = building · under construction
dashed violet line = intended dependency
requested · grilling · grilled pills = a candidate being grilled
What each metric means
Depth higher = better
How much logic is hidden inside vs leaking out. A deep module does a lot of work behind a small, simple surface — callers don't need to care how it works. A shallow module is basically a wrapper that exposes everything.
Coverage higher = better
How much of the module is tested. 0% = nothing is tested, 100% = everything is tested. Low coverage on an important module is the most dangerous combination — you can't safely change it.
Leak always bad
A module is reaching into something it shouldn't depend on. Like a checkout page importing database internals directly. Every leak is a hidden coupling — change one thing and something far away breaks.
Proposal
The agent spotted a way to improve a module — tighten its interface, split it up, or fix a bad dependency. It's not a bug report, it's a suggestion. You decide: accept it, defer it, or reject it.
Grilling
Putting a proposal under the microscope. The agent digs into whether it's actually worth doing — cost vs benefit. Ends with a verdict: yes, not now, or no.
Updated needs attention
The code changed since the last scan. The module's depth/coverage info might be out of date — worth re-scanning to see if the numbers still hold.
Orphan worth investigating
Nothing connects to this module and it connects to nothing. It's either dead code, a very early stub, or something that was added to the map by mistake.
Planned
A module that doesn't exist in the code yet — it's the intended future design. Shows as a dashed ghost card. Once it gets built, the ghost becomes a real card.
Domain
A group of related modules — like "api", "frontend", "seeds". Just a way to cluster things visually. Cross-domain arrows are the seams between subsystems.

Work with the agent, not just the code

arch-map shows your repo as a living dependency graph. An agent watches it and proposes where to deepen, split, or fix the architecture. Your job is to decide — and everything stays in sync as you go.

1
The graph is the map
Every card is a module. Position and fill show how deep it sits; the bar shows test coverage. Hover to light up its neighbourhood, click to inspect.
2
The rail is the agent
On the right: its proposals queue. Each one names a problem, a solution, and the wins. Hover a proposal to find it on the graph. The agent also drafts plans — the intended structure — in the Plans tab; planned modules show as dashed ghosts on the graph.
3
You close the loop
Accept, defer, reject or dismiss — with a reason. The graph, counts and queue update instantly. That's the whole game.
4
Find the trouble fast
The ⚠ Issues panel (left edge or p) ranks every structural problem. Toggle Overview ↔ Detail to zoom from domains to modules. Click any card → the Inspector shows its graph metrics and signals (leaky seam, god module, bottleneck…).
